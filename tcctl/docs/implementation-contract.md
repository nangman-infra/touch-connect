# tcctl Implementation Contract

## Purpose

`tcctl` is the operator and admin CLI for touch-connect.

It is a human control surface.
It does not execute AI work and does not replace `tc-worker`.

## Runtime Role

`tcctl` is a short-lived command process.

It owns:

- operator commands
- task and message creation requests
- state inspection
- approval decisions
- retry and DLQ replay requests
- canonical scenario development commands

It does not own:

- endpoint registration automation
- message processing
- local workspace execution
- worker heartbeat
- checkpoint generation
- source of truth state

## Command Groups

Minimum command groups:

```text
tcctl server
tcctl endpoint
tcctl task
tcctl message
tcctl artifact
tcctl approval
tcctl dlq
tcctl scenario
```

The `server` command group is a CLI UX label for platform health/version inspection.
It still calls `tc-control`; it does not bypass the control plane.

## Required Commands

### server

```text
tcctl server health
tcctl server version
```

### endpoint

```text
tcctl endpoint list
tcctl endpoint inspect <endpoint_ref>
tcctl endpoint capabilities
```

### task

```text
tcctl task create
tcctl task status <task_id>
tcctl task history <task_id>
tcctl task retry <task_id>
tcctl task cancel <task_id>
```

### message

```text
tcctl message send
tcctl message inspect <message_id>
tcctl message history --task <task_id>
```

### artifact

```text
tcctl artifact list --task <task_id>
tcctl artifact inspect <artifact_version_ref>
tcctl artifact finalize <artifact_version_ref>
```

### approval

```text
tcctl approval list
tcctl approval inspect <approval_id>
tcctl approval approve <approval_id>
tcctl approval reject <approval_id>
```

### dlq

```text
tcctl dlq list
tcctl dlq inspect <dead_letter_id>
tcctl dlq replay <dead_letter_id>
```

### scenario

```text
tcctl scenario run canonical
tcctl scenario verify canonical
```

## API Contract

`tcctl` communicates with `tc-control` through the Control API.

Rules:

- verify compatible Control API version before executing commands that depend on versioned schemas
- no direct writes to storage
- no direct writes to JetStream
- no local shell execution for task work
- no direct mutation of server-owned state
- command output must reflect `tc-control` accepted state backed by server records or projections, not local assumptions

## Input Contract

CLI commands that create messages must map to the canonical message envelope.

Required fields for message creation:

```text
target_capability
payload.summary
payload.body
payload.references[]
constraints[]
delivery_class
readback_required
```

If protected side effect intent is included, `idempotency_key` and protected scope fields must be supplied or generated by server policy.

## Approval Contract

Approval commands must preserve human accountability.

Rules:

- `approve` and `reject` require actor identity
- self-approval must be rejected unless policy explicitly allows it
- approval decision does not mutate original message
- approval decision must include current `approval_hash`
- expired approval requests cannot be approved

## Output Contract

Default output should be stable and machine-readable when requested.

Required modes:

- human-readable table or text
- JSON output with `--json`

Human output may be concise, but JSON output must preserve exact ids and refs.

## Configuration

Minimum configuration keys:

```text
control_url
actor_id
workspace_id
output_format
request_timeout
auth_token
config_file
```

Configuration sources, in priority order:

1. command flags
2. environment variables
3. local config file
4. built-in defaults

## Exit Codes

Required exit code classes:

- `0`
  - success
- `1`
  - command or validation failure
- `2`
  - server rejected the request by domain contract
- `3`
  - transport or server unavailable
- `4`
  - authentication or authorization failure

## Canonical Scenario Contract

`tcctl scenario run canonical` is a development helper for the MVP canonical flow.

It may create:

- room
- primary thread
- task
- initial handoff message
- approval decision when explicitly requested

It must not fake worker checkpoints unless a fake-worker mode is explicitly enabled.

`tcctl scenario verify canonical` checks:

- at least one readback-required handoff succeeded
- at least two artifact versions exist
- validation report includes SonarQube gate pass
- approval decision is approved
- side effect execution record is succeeded
- side effect uniqueness boundary executed once
- final task state is completed

## Test Contract

Minimum tests:

- command parsing
- control API request mapping
- Control API compatibility failure
- canonical message creation payload
- JSON output stability
- approval decision payload
- artifact finalization payload
- server/control rejection handling
- unavailable control plane handling
- canonical scenario verification logic
